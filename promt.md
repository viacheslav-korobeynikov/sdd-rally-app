
Проект: веб‑приложение для организаторов автомобильных ралли 3 категории (регистрация соревнований, управление спецучастками, заявками экипажей, классификацией результатов).
Цель: единая платформа для подготовки и проведения ралли, минимизации ручной работы организаторов и снижения рисков ошибок в протоколах.

Нефункциональные принципы и архитектура (запиши как «Core Principles» с обоснованием для каждого пункта):

Архитектурный стиль: модульный монолит с чётким разделением доменов (соревнования, маршруты/спецучастки, экипажи и участники, результаты). Разделить код и слои так, чтобы в будущем можно было выделить части в отдельные сервисы без больших переписываний.

Backend: один сервис на Golang (Fiber) с REST/HTTP API. Обязательны: чистая архитектура (слои handler/service/repository), контекст для запросов, middleware для логирования, трейсинга и аутентификации, недопускается прямое обращение к базе данных из обработчиков запросов.

Frontend: серверный рендеринг на Templ + HTMX. Максимум логики оставлять на backend, HTMX использовать для интерактивных обновлений (таблицы заявок, протоколы, фильтры). Не использовать SPA‑фреймворки.

База данных: PostgreSQL как единственный источник истины. Строгая нормализация основных доменных сущностей (соревнование, спецучасток, экипаж, пилот, автомобиль, результаты, штрафы). Жёсткий запрет на бизнес‑логику в триггерах и хранимых процедурах, кроме технических / audit‑кейсов.

Авторизация и аутентификация: login/password, хранящиеся в БД в виде bcrypt/argon2‑хеша. Обязательны: минимально необходимый набор ролей (например, «Главный организатор», «Секретарь», «Хронометраж», «Наблюдатель/чтение»), принцип минимальных привилегий, запрет хранения секретов в репозитории.

Domain‑driven подход: явные доменные модели под специфику ралли (классы автомобилей, группы, зачёты, стартовые ведомости, протоколы результатов, штрафы за опоздание/опережение и т.д.). Все бизнес‑правила по регламенту ралли оформлять в коде и/или отдельной документации.

Наблюдаемость и качество: структурированные логи (JSON), базовые метрики (кол-во заявок, время генерации отчётов, ошибки API). Обязательные unit‑тесты для критичных доменных сервисов (например, расчёт результатов и штрафов).

Семантическое версионирование: использовать MAJOR.MINOR.PATCH для релизов платформы, фиксировать изменения в changelog.

Безопасность: защита от типичных веб‑уязвимостей (CSRF, XSS, SQL‑инъекции), использование параметризованных запросов к БД, централизованная настройка CORS.

Стандарты разработки и процесса (оформи отдельным разделом «Development Standards»):

Git‑процесс: ветки по фичам, обязательный code review, линтеры (golangci‑lint или аналог).

Документация: для backend — README, диаграмма основных доменных сущностей, описание основных API‑эндпоинтов; для frontend — схема основных экранов (регистрация, управление СУ, списки экипажей, протоколы).

Миграции БД: использовать миграции (например, golang-migrate), хранить их в репозитории, запрещены ручные изменения схемы на проде.

Инфраструктура и окружения (раздел «Infrastructure & Environments»):

Локальная разработка: docker-compose (Golang backend, PostgreSQL, вспомогательные инструменты).

Staging/Prod: контейнеры Docker, с прицелом на развёртывание в Kubernetes в будущем. Разделение конфигурации по окружениям через переменные окружения и secrets.
